# VARCHAR 字段长度设计规范

> 适用数据库:MySQL 8.x(默认字符集 utf8mb4)

## 一、核心结论

- `varchar(N)` 中的 **N 是字符数,不是字节数**(MySQL 5.0.3 起即如此)。一个中文、一个英文字母、一个数字、一个希腊字母都各算 1 个字符。
- VARCHAR 是**变长存储**:`varchar(20)` 和 `varchar(255)` 存同样的内容,占用字节数完全一样。N 只是一个「最大长度约束」。
- **不要反射性地把字段一律设成 `varchar(255)`**。应按业务真实上限,留 1.5~2 倍余量取值(例如「约 20 字符」的字段设 `varchar(50)`)。

## 二、背景:为什么不建议「为了扩展性一律设 255」

一种常见想法是:程序层(前后端)已限制输入为 20 字符,但担心「上线后改数据库很麻烦」,于是把列设成 `varchar(255)` 留余量。

这个想法依赖的前提——「上线后改不动」——在 MySQL 8 里基本是个误解。

### 2.1 VARCHAR「加长」在 MySQL 8 是廉价的在线操作

把长度约束**改大**,通常是在线、不锁表、不拷贝整表的:

```sql
-- 在线完成,允许并发读写,基本瞬间返回
ALTER TABLE t MODIFY name varchar(50) NOT NULL, ALGORITHM=INPLACE, LOCK=NONE;
```

唯一的限制是**长度前缀字节数不能变**:

| 字符集 | 1 字节前缀范围 | 2 字节前缀范围 |
| --- | --- | --- |
| utf8mb4 | 最大字节数 ≤ 255,即 `varchar(0~63)` | `varchar(64)` 及以上 |

- `varchar(20)` → `varchar(50)`:都在 1 字节前缀区,`ALGORITHM=INPLACE`,在线完成。
- 从「≤ 63 字符」跨到「≥ 64 字符」:前缀从 1 字节变 2 字节,才需要拷贝整表。

> 真正麻烦的是**改小**(需 `ALGORITHM=COPY` 拷贝整表并校验)。而我们担心的「将来不够用要扩大」恰好是廉价方向。

结论:用「上线后没法改」当作设 255 的理由,并不成立。

## 三、设成 255 实际放弃了什么

### 3.1 丢掉数据库这道最后的完整性防线(最重要)

应用层校验可以被绕过:前端被改、新接入的客户端、内部管理后台、数据导入脚本、另一个直连库的服务、手动 `UPDATE` 等。这些场景下前后端的「20 字符」限制都不生效,而**数据库列约束是最后一道闸**。

- 设 `varchar(20)`:超长数据被库直接拒绝,坏数据进不来。
- 设 `varchar(255)`:可能悄悄写入一条 200 字符的脏数据,发现时已在生产库。

这是「纵深防御」,也是列长度约束最有价值的作用。

### 3.2 若字段要建索引,255 更吃亏

utf8mb4 下 `varchar(255)` = 1020 字节。InnoDB 单索引键前缀上限 3072 字节(DYNAMIC 行格式),单列勉强够,但**联合索引**里放几个这种列就可能超限。`varchar(20)`(80 字节)做索引则毫无压力。

### 3.3 schema 即文档

`varchar(20)` 直接表达「这是个短名字字段」;`varchar(255)` 等于没表达意图。

## 四、平衡:255 在什么情况下确实无害

MySQL 8.0 削弱了过去反对「超长 VARCHAR」的两个经典理由:

- 内部临时表默认引擎改为 **TempTable**,VARCHAR 按实际长度存储,不再按声明最大值分配定长行;
- 排序(filesort)采用 **packed addon fields**,排序缓冲区按实际长度打包。

因此,若字段**既不建索引、也不参与排序/分组**,只是普通字段,设成 255 在性能与存储上几乎无副作用。这种情况下用 255 不算错,只是收益不大。

## 五、落地规范

| 字段用途 | 建议长度 |
| --- | --- |
| 普通字段,业务约 N 字符,需留余量 | 取 `varchar(N×1.5~2)`,且尽量 ≤ 63(落在 1 字节前缀区)。如「约 20 字符」→ `varchar(50)` |
| 该字段要建索引 / 频繁查询排序 | 按真实上限取紧,如 `varchar(20~32)` |
| 纯普通字段、不索引、图省事 | `varchar(255)` 可接受,但非首选 |

### 字符集要求

要同时容纳中文 / 英文 / 数字 / 希腊字母等,列/表字符集必须为 **utf8mb4**(MySQL 8 默认)。`latin1` 等无法存储中文与希腊字母。

```sql
`name` varchar(50) CHARACTER SET utf8mb4 NOT NULL
```

## 六、一句话总结

**别把「上线后没法改」当成设 255 的理由(加长本就不难),而应按业务真实上限留 1.5~2 倍余量取值,既保留数据库这道校验闸,又留足扩展空间。** 对「约 20 字符」的字段,推荐 `varchar(50)`。
